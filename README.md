# Конструктор запросов UN Comtrade

Статическая русскоязычная версия конструктора и аналитического дашборда для
GitHub Pages. Сборка рассчитана на работу без VPN с российских IP.

## Почему нужен отдельный прокси

GitHub Pages не выполняет серверный код, а официальный UN Comtrade API не
разрешает надёжные прямые запросы из браузера. В прежней версии использовался
домен Cloudflare Workers, который может быть недоступен или сильно замедляться
в России. В этой версии Cloudflare полностью исключён: запросы идут через вашу
публичную функцию в Yandex Cloud.

Рабочая цепочка:

`GitHub Pages → Yandex Cloud Function → comtradeapi.un.org`

## 1. Публикация функции в Yandex Cloud

1. Откройте консоль Yandex Cloud и выберите или создайте каталог.
2. Откройте **Cloud Functions → Создать функцию**.
3. Задайте имя, например `un-comtrade-proxy`, и создайте функцию.
4. Создайте новую версию со следующими параметрами:
   - среда выполнения: **Node.js 22**;
   - способ: редактор кода;
   - точка входа: `index.handler`;
   - тайм-аут: **60 секунд**;
   - память: **256 МБ**.
5. Загрузите готовый архив `deploy/un-comtrade-yandex-function.zip` либо два
   файла из папки `yandex-cloud-function`: `index.js` и `package.json`.
6. Создайте версию и включите **публичный доступ** к функции.
7. Скопируйте URL вызова вида
   `https://functions.yandexcloud.net/идентификатор-функции`.
8. Откройте URL в браузере без VPN. Должен появиться ответ:
   `{"ok":true,"service":"UN Comtrade proxy","maxRecords":2000}`.

Функция не хранит API-ключ и не использует cookie. Ключ передаётся только в
текущем запросе к официальному API.

## 2. Обновление GitHub Pages

1. Загрузите всё содержимое этой папки в ветку `main` вашего репозитория.
2. Откройте **Settings → Secrets and variables → Actions → Variables**.
3. Измените переменную `NEXT_PUBLIC_COMTRADE_PROXY_URL`: укажите полный URL
   функции Yandex Cloud без дополнительного пути в конце.
4. Откройте **Settings → Pages** и убедитесь, что источник — **GitHub Actions**.
5. Во вкладке **Actions** повторно запустите workflow
   **Deploy to GitHub Pages**.
6. После успешной публикации обновите страницу через `Ctrl+F5`.

Workflow `.github/workflows/pages.yml` сам учитывает подпапку репозитория и
поддерживает адреса как `https://имя.github.io/`, так и
`https://имя.github.io/репозиторий/`.

## Что изменено для стабильной работы

- исключены Cloudflare Workers и домены `workers.dev`;
- не используются ресурсы ChatGPT/OpenAI;
- прокси и браузер ограничивают время ожидания, поэтому вечной загрузки нет;
- одна часть очереди ограничена 2 000 строками, чтобы ответы стабильно проходили
  через HTTP-функцию;
- крупные выборки автоматически дробятся и выполняются строго последовательно;
- ответы объединяются в браузере для графиков и выгрузки CSV;
- CORS `Access-Control-Allow-Origin: *` возвращается для POST, OPTIONS, ошибок и
  проверки доступности.

## Локальная проверка

Требуется Node.js 22.

```bash
npm ci
npm run test:proxy
npm run build:pages
npx serve out
```

После сборки готовый статический сайт находится в папке `out`.

## Основные команды

- `npm run build:pages` — статическая сборка для GitHub Pages;
- `npm run test:proxy` — проверка CORS и ответов функции;
- `npm run lint` — проверка исходного кода;
- `npm run dev` — локальный режим исходного проекта.
